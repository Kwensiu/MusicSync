# MusicSync 方案说明

## 1. 项目定位

MusicSync 是一个面向 Windows 和 Android 的局域网音乐目录同步工具。第一版目标不是做完整双向同步，而是优先做一个安全、可解释、能稳定落地的单向镜像同步工具。

第一版采用 `A -> B` 镜像模型：

- A 作为源端
- B 作为目标端
- 用户先预览差异，再决定是否执行同步
- 可选删除目标端多余文件
- 同名但内容不同的文件统一标记为冲突，不自动覆盖

当前工程实现已进一步收敛为：

- 同步语义固定为 `本机 -> 远端`
- 当前优先承诺的主路径是 `Windows -> Android`
- `Android -> Windows` 仍处于兼容性修复阶段

该工具的主要适用场景是：

- Windows 电脑将音乐目录同步到 Android 设备
- Android 设备将音乐目录同步到 Windows 电脑
- 两台设备在同一局域网内，通过手动输入地址建立连接

第一版不依赖 USB/MTP，不做云同步，不做后台实时监听。

第一版还应明确接受以下运行约束：

- Android 端切到后台后，系统可能回收网络连接
- 因此当前局域网同步要求目标设备保持前台
- 当前传输协议仍以“先跑通”为目标，性能优化放在稳定性之后

## 2. 已确认的产品取舍

当前方案基于以下已确认决策：

- 框架选择：Flutter
- 同步模式：仅支持 `A -> B` 镜像
- 跨设备方式：Windows 与 Android 通过局域网直连
- 首版连接方式：手动输入地址连接
- 自动发现设备：作为后续 TODO
- 差异判断：快速模式
- 缓存策略：简单缓存
- 冲突策略：同名不同内容一律标记冲突，不自动覆盖
- 删除策略：镜像模式下支持可选删除，默认关闭

这些取舍的核心目的是控制第一版复杂度，同时保证同步行为足够清晰和安全。

## 3. 为什么选 Flutter

Flutter 适合作为这一版的客户端框架，原因主要有三点：

- Windows 和 Android 都能共用一套 UI 和大部分业务逻辑
- 构建桌面与移动端界面的效率较高
- 便于后续扩展其他平台

但方案不应把 Flutter 理解为“纯 Dart 即可解决所有文件系统问题”。在 Windows 上，本地目录访问相对直接；在 Android 上，目录授权、目录遍历、写入与删除必须通过平台允许的方式实现。因此，本项目的合理架构应是：

- Flutter 负责 UI 与业务流程
- Dart 负责扫描、差异分析、同步计划与网络协议
- Android 文件访问通过平台适配层处理

## 4. 产品范围

第一版支持以下能力：

- 选择本机音乐目录
- 手动输入远端设备地址并建立局域网连接
- 读取源端与目标端目录索引
- 按相对路径、文件大小、修改时间进行快速对比
- 生成同步预览
- 执行单向镜像同步
- 可选删除目标端多余文件
- 显示进度、日志、错误信息
- 支持取消同步任务
- 对冲突文件单独列出，不自动处理
- 使用简单缓存提升重复扫描性能

当前实现已经对 UI 流程做过一次收敛调整：

- 首页已升级为单页工作台
- 源目录选择、目标目录选择、差异预览、执行进度被合并到同一工作区
- 原预览页与执行页暂时保留为过渡结构，但主流程不再依赖多页跳转
- 预览始终展示完整差异结果
- 删除风险不再通过预览开关控制，而是在执行前通过确认弹窗处理

这样做的原因是该工具本质上是单次任务工作台，而不是多步骤表单。对这类同步工具而言，把“选择目录、查看差异、立即同步”放在一个页面中更符合使用预期，也能减少状态在多页之间来回切换带来的复杂度。

删除交互也已同步调整为：

- 预览阶段始终如实展示复制项、删除项、冲突项
- 执行阶段若包含删除项，则要求用户明确确认

这样可以避免“为了安全隐藏真实差异”的反模式，同时把风险控制放在真正发生破坏性操作的时刻。

第一版明确不支持：

- 完整双向同步
- 自动冲突解决
- 自动发现局域网设备
- 云同步
- 后台实时监听
- 断点续传
- 全量哈希严格校验

## 5. 同步规则

同步方向固定为 `A -> B`。

默认比较依据为：

- 相对路径
- 文件大小
- 修改时间

差异规则定义如下：

- 仅 A 存在：加入复制计划
- 仅 B 存在：
  - 如果用户开启删除选项，加入删除计划
  - 如果未开启删除，忽略
- A 和 B 都存在且大小、修改时间一致：视为一致，跳过
- A 和 B 都存在但元数据不一致：标记为冲突，不自动覆盖

该规则刻意避免在第一版中自动判断“谁更新、谁应覆盖”，从而降低误删和误覆盖风险。

## 6. 整体架构

系统建议划分为四层：

### 6.1 表现层

Flutter UI 负责：

- 本地目录选择
- 目标目录选择
- 设备地址输入
- 连接状态展示
- 差异列表展示
- 同步预览
- 同步执行控制
- 进度与日志显示

### 6.2 业务层

纯 Dart 实现核心逻辑：

- 目录扫描
- 文件索引构建
- 差异分析
- 同步计划生成
- 任务调度
- 进度回调
- 取消控制
- 缓存管理

### 6.3 平台文件访问层

平台差异不应散落在业务逻辑中，而应统一封装。建议抽象以下接口：

- 选择目录
- 列举目录内容
- 读取文件元数据
- 打开读取流
- 打开写入流
- 创建目录
- 删除文件或目录

实现上至少分为：

- Windows 文件访问实现
- Android 文件访问实现

### 6.4 网络传输层

第一版采用局域网 TCP 直连与自定义轻量协议：

- 控制消息负责握手、索引交换、任务控制
- 文件内容通过流式传输发送
- 连接建立方式为手动输入地址

相较于 WebSocket，TCP 更适合大文件连续传输和传输进度控制。

补充说明：

- 当前并未使用 `HTTP` 或 `HTTPS`
- 当前传输慢并不是由 HTTPS 导致
- 当前性能瓶颈主要来自：
  - 自定义控制消息中的 `JSON + base64`
  - Flutter 与 Android 原生之间的 `MethodChannel` 分块读写
  - 当前仍为顺序传输而非高吞吐优化版协议

## 7. 通信模型

建议采用目标端监听、源端发起连接的模型：

- B 端进入等待接收状态并启动监听服务
- A 端输入 B 的地址后发起连接
- 双方建立控制连接
- 交换目录索引
- A 端根据双方索引生成同步计划
- 用户确认后开始执行复制和可选删除

这样做的优点是：

- 同步方向明确
- 删除语义明确
- 任务控制集中
- 未来增加自动发现功能时只需改进连接建立过程

## 8. 数据模型建议

建议至少定义以下核心对象：

### 8.1 FileEntry

描述单个文件或目录：

- `relativePath`
- `isDirectory`
- `size`
- `modifiedTime`
- `fingerprint`，可空

### 8.2 ScanSnapshot

表示某次扫描结果：

- 根目录标识
- 文件条目集合
- 扫描时间
- 缓存信息
- 扫描警告列表

这里的“扫描警告列表”用于承载“根目录可访问，但部分子目录不可访问”的情况。当前实现约定为：

- 根目录不可访问：本次扫描直接失败
- 子目录不可访问：记录为 warning，并允许后续继续生成预览

这样可以避免把“部分成功”伪装成“完全成功”，也能避免因为单个异常子目录导致整个预览不可用

### 8.3 DiffItem

表示单个差异项：

- `copy`
- `delete`
- `conflict`
- `skip`

### 8.4 SyncPlan

表示本次同步计划：

- 待复制列表
- 待删除列表
- 冲突列表
- 统计摘要

### 8.5 TransferProgress

用于同步过程反馈：

- 当前阶段
- 当前文件
- 已处理文件数
- 总文件数
- 已传输字节
- 总字节数

## 9. 缓存策略

第一版只做简单缓存，不引入复杂数据库设计。

缓存建议记录：

- 相对路径
- 文件大小
- 修改时间
- 上次扫描时间
- 可选指纹字段

缓存主要用途：

- 加快重复扫描
- 减少重复读取元数据
- 为后续严格模式预留接口

缓存失效条件可以简单定义为：

- 文件大小变化
- 修改时间变化
- 路径新增或删除

## 10. Android 实现边界

Android 端不应按桌面文件系统思路设计。第一版必须明确接受以下事实：

- 用户需要主动授予目录访问权限
- 应用需要保存并复用该权限
- 目录遍历、读取、写入、删除需通过平台支持的目录访问方式完成

因此 Android 端的关键不只是“选目录”，而是“安全且稳定地访问用户授权目录”。这是本项目的重要实现前提。

## 11. 风险点

第一版主要风险包括：

- Android 目录访问实现复杂度高于 Windows
- 局域网连接受不同网络环境影响，需处理断连和超时
- 大文件传输过程需要稳定的进度反馈和取消机制
- 删除操作必须有清晰预览和确认，避免误删
- 快速模式可能无法覆盖所有“内容变更但时间戳异常”的边缘情况

其中风险最高的是 Android 平台适配与局域网同步稳定性。

## 12. 后续演进方向

当前方案为一条可落地的首版路线，后续可逐步扩展：

- 自动发现局域网设备
- 严格模式或按需哈希校验
- 冲突处理策略增强
- 断点续传
- 同步历史与执行报告归档
- 更细粒度的过滤规则

同时，当前已知仍需继续补全的关键点包括：

- 本地双目录真实差异对比
  - 当前本地执行已经可跑通
  - 但预览目标目录必须真正接入扫描与差异计算，不能继续使用空目标占位
- 本地执行器的删除策略
- 远端目录扫描与索引交换

## 13. 结论

该方案的核心价值在于明确边界：

- 不做完整双向同步
- 不依赖 MTP
- 不把 Android 当桌面文件系统
- 不在第一版自动处理冲突

在这些边界下，采用 Flutter 构建 Windows 和 Android 客户端，并通过局域网直连实现 `A -> B` 镜像同步，是一条务实且可以真正落地的路线。

## 14. 当前实现补充

最近已补充两条与执行稳定性直接相关的行为：

- 传输中的目标文件先写入 `*.music_sync_tmp`，完成后再重命名为正式文件
- 工作台执行区已提供“停止同步”按钮

当前停止语义为：

- 目标是尽快停止，而不是回滚已经完成的复制或删除
- 本地复制中断时，会删除当前未完成的临时文件
- 远端复制中断时，会通过 `abortCopy` 清理远端临时文件
- 如果目标端被系统冷杀，应用无法仅凭内存态识别残留临时文件，因此当前采用保守策略：
  - 扫描默认忽略 `*.music_sync_tmp`
  - 首页提供手动清理残留入口

另外，执行完成或手动停止后，首页会自动重建当前预览：

- 本地调试模式会重新扫描本地目标目录
- 远端模式会先刷新远端索引，再重建远端预览

这一步的目的不是追求“无感刷新”，而是避免同步后仍然停留在旧计划上，造成用户误判。

## 15. 里程碑基线

首个里程碑 commit 完成后，可以把当前工程理解为：

- 一个以首页工作台为中心的 Flutter 同步工具
- 已具备 Windows 与 Android 的首版局域网同步闭环
- 已把“半成品文件泄露为正式文件”的风险收口到临时文件策略
- 已把“手动停止同步”和“同步后仍显示旧计划”的行为收口到当前状态模型

当前下一轮开发不应再回头重做基础链路，而应建立在这个基线上继续优化体验、性能和兼容性。

当前第二轮已开始做低风险体验优化，主要包括：

- 差异列表轻量化渲染
- 首页与结果页的执行状态语义增强
- 远端索引时间与当前预览目标快照时间展示

这些改动的目标是让后续人工测试更容易判断“当前看到的是不是最新状态”，同时优先改善大量差异项场景下的可用性。

最近又补充了一轮首页引导与错误收口：

- 首页新增“当前流程”区块，用于把当前会话拆成可观察的几个阶段
  - 本地目录是否已准备
  - 远端连接是否已建立
  - 远端目录索引是否已可用
  - 当前是否已有可执行的远端预览
- 系统会基于当前状态给出单条“下一步”提示
- 这使首页更接近任务工作台，也减少了“监听 / 连接 / 预览 / 执行”之间的语义混淆
- 剩余底层异常文本也继续被归一化，以避免执行区和预览区再次出现中英混杂

基于真实联调流程，首页又继续向“顺序化单向同步工作流”收口：

- 正式主路径不再以多个平级卡片并列展示
- 当前更强调以下顺序：
  1. 连接远端设备
  2. 选择本地源目录
  3. 自动生成远端预览
  4. 执行远端同步
- 本地调试目标与本地复制链路仍保留，但已被降级到“高级与调试”区域

这一调整背后的核心判断是：

- 当前产品不是双向同步，也不是对等双端操作台
- 因此 UI 不应继续制造“本地 / 远端两边是同级对象”的错觉
- 在第一版里，首页应该优先服务“把本机内容同步到远端”这一条明确流程

设置体系也已开始落地第一项正式配置：

- 启动时自动开启监听
- 这项设置只控制应用启动后的默认网络待机状态
- 它不改变 `本机 -> 远端` 的同步语义
- 因而可以被视作独立的运行时偏好，而不是某种新的传输模式

最近还补上了一条关键的状态语义修正：

- 连接状态与远端目录是否已准备不再被视作同一件事
- 连接建立后，如果远端目录尚未选择，共享会话仍然保留
- 远端之后再选择目录时，会通过现有会话向对端发送“目录状态已变化”的轻量通知

这使得首页主流程终于更接近用户真实理解：

- 先连上设备
- 再等待或完成两端目录准备
- 准备好后继续索引与预览

而不是反过来通过一次“连接失败”来提示用户缺少前置条件。

在首页表现层上，也进一步把“目录状态”和“索引状态”拆开表达：

- 远端目录未选择
- 远端目录已选择但索引仍在刷新
- 远端索引已可用

这类细分虽然不改变底层逻辑，但能显著降低用户在联调时的误判成本。

截至当前版本，步骤 1“连接远端设备”也已经收口为正式产品形态：

- 首页首屏不再承担调试工作台职责，而是优先服务“先连上设备”这一真实起点
- 步骤 1 当前只保留五类信息：
  - 连接状态胶囊
  - 监听端口胶囊
  - 分享地址入口
  - 地址输入框
  - 连接 / 停止连接按钮
- 分享地址弹窗已改为轻量 `Dialog`
  - 使用二维码 + 地址卡片的双通道展示
  - 地址区域按更接近 Material code block 的样式表达“这是一段可复制连接信息”
  - 分享内容固定指向本机监听地址，不再混入对端地址语义

这意味着首页前半段的信息架构已经从“开发者工作台”转为“面向普通用户的连接入口”，后续重心应转到步骤 2 及其后的目录、预览和执行流程，而不是继续在连接区叠加更多信息。

步骤 2“选择本地源目录”目前也已形成相对稳定的结构：

- 主体采用单张源目录状态卡片
- 最近目录与最近地址共享同一套“历史记录管理”思路
  - 首页保留轻量快捷入口
  - 深度管理放进独立弹窗
- 历史记录管理当前支持：
  - 拖拽排序
  - 编辑备注
  - 删除
  - 最近地址额外支持直接编辑地址本身

这意味着步骤 2 不再只是“目录选择按钮”，而是已经开始承担“源目录状态管理”和“高频历史入口整理”的职责。不过它仍保持在单页工作台内完成，没有重新拆回多页面流程。

随后首页主流程又继续做了一次关键收口：

- 原本独立的“差异预览”和“执行同步”已合并成同一张步骤 3 卡片
- 卡片内部按“分析状态 -> 预览摘要 -> 差异明细 -> 执行状态 -> 执行动作”顺序展开
- 执行按钮不再需要用户在预览之后再切换到另一块几乎重复的上下文里寻找

这次合并背后的判断是：

- 当前产品仍然是单向镜像同步，不是复杂编排工具
- 用户真正关心的是“现在会同步什么”和“要不要立刻执行”
- 因此把预览和执行拆成两个并列步骤，会制造流程感，却不会增加真正的信息价值

同一轮收口里，步骤 1 也去掉了“远端共享目录是否已就绪”这类细碎状态提示。原因不是这类状态不重要，而是它已经可以通过连接状态、自动分析状态和错误提示自然传达，不需要在首屏额外占一行动态文案。

截至当前版本，首页的信息架构可以概括为：

- 步骤 1：建立连接
- 步骤 2：确认源目录
- 步骤 3：检查并同步

这使首页终于从“开发期工作台”进一步收束成“可发布首版交互”，也给后续继续做自动发现、设置项收口和主题统一留下了稳定骨架。

步骤 3 最近又做了一轮明显偏产品层的收口：

- 预览摘要卡已被移除，避免与下方类型筛选和执行状态重复表达
- 项目类型与文件类型筛选都改成了更贴近实际使用的多选模型
- `全部项目` 与 `全部类型` 都被处理成真正的总览态，而不是简单等价于“把所有具体项都勾上”
- 筛选逻辑被收回到首页前端层，不再因为切换筛选就触发预览重建

这一步的重要性在于，它重新划清了两类职责：

- 预览构建：决定“计划里到底有什么”
- 页面筛选：决定“用户此刻想看计划里的哪一部分”

同一轮里，差异列表的视觉边界也被收口：

- 列表外层新增单独容器，和首页卡片形成第二层视觉分组
- 列表项本身再形成第三层轻量分组
- 没有结果时，列表容器不会消失，而是在容器内部显示空状态

这让步骤 3 的阅读体验更稳定，不会因为筛选条件变化而在结构上忽胖忽瘦。

设置体系最近也开始承担真正的“全局规则”职责，而不再只放运行时开关：

- 新增 `忽略后缀`
- 这组后缀会在预览和同步计划构建时直接生效
- 因此它和首页里的“文件类型筛选”不是同一层东西

当前产品语义已经明确分层为：

- 设置页 `忽略后缀`：全局规则，决定哪些内容不进入计划
- 首页 `文件类型`：页面筛选，决定当前怎样查看已生成的计划

为了让这层关系对用户可见，步骤 3 还补了一条轻量摘要：

- 当存在忽略规则时，会在“文件类型”标题右侧显示 `已忽略：.x, .x`

这类做法虽然很小，但对同步工具特别重要，因为它能减少“为什么某个文件不见了”这类典型困惑。

# 远端冲突项 Tag 元数据补丁方案

本文档记录当前关于“步骤 3 是否可从仅查看目标端文件 tag，扩展为允许在设备 A 对设备 B 的文件 Tag 元数据进行补写”的讨论结论。本文只讨论方案，不涉及当前代码实现。

## 1. 结论

该能力可以做，而且方向合理；但建议将其定义为：

- 设备 A 生成针对设备 B 文件的元数据补丁请求
- 设备 B 的 MusicSync 接收请求后
- 由设备 B 本机执行实际的 tag 写入

不建议在概念上定义为“设备 A 直接编辑设备 B 的文件”。更准确的表述应当是：

- 远端元数据补丁（metadata patch）
- 目标端代理执行写入

这样更符合协议边界、安全边界与错误处理模型。

## 2. 为什么适合从步骤 3 继续扩展

当前步骤 3 已经具备以下前提：

1. 设备 A 能识别设备 B 上的目标文件
2. 设备 A 能看到设备 B 的 tag 信息
3. 用户可以基于两端元数据差异，对冲突项做判断

因此，后续自然可以扩展为：

1. 用户在设备 A 上完成冲突决策
2. 设备 A 生成一份对设备 B 的 metadata patch
3. 设备 B 收到请求后，在本机写回 tag
4. 写回成功后刷新目标项状态

对于“冲突项”，这种模型比重传整个音频文件更合适。

## 3. 为什么优先做 metadata patch，而不是重传文件

### 3.1 传输成本更低

冲突项多数情况下是：

- 文件内容相同，但 tag 不同
- 一侧 tag 更完整
- 双方不同字段各有优劣

此时只传输元数据补丁即可，无需重传音频文件本体。

### 3.2 冲突处理更细粒度

冲突解决往往不是“整文件选 A 或选 B”，而是字段级决策，例如：

- 保留 A 的 `title`
- 保留 B 的 `albumArtist`
- 合并 `genre`
- 清空某一侧错误 `comment`

因此更适合设计为字段级 patch，而不是文件级覆盖。

### 3.3 架构边界更清晰

职责应当拆分为：

- 设备 A：负责展示差异、收集用户决策、生成 patch
- 设备 B：负责鉴权、校验文件状态、执行写入、返回结果

这样比“远程直接操作对方文件系统”更安全，也更容易约束能力边界。

## 4. 建议的能力定义

建议新增的不是“通用远程编辑文件能力”，而是：

- 面向远端媒体文件的 tag 元数据补丁能力
- 首阶段仅服务于冲突项
- 目标端执行写入

也就是说，设备 A 提交的是“修改意图”，不是“文件系统级控制命令”。

## 5. 核心模型建议

## 5.1 目标文件定位

A 必须能够稳定定位 B 上的目标文件。只靠路径不够稳妥，建议至少具备以下信息中的一部分：

- remote device id
- file id / library item id
- relative path
- file size
- modified time
- content hash
- metadata fingerprint / tag revision

原因是仅靠路径会遇到以下问题：

- 文件已被移动或重命名
- 路径相同但文件内容已变
- 预览时看到的是旧状态，提交时文件已更新

因此，路径可以作为定位信息的一部分，但不应当是唯一主键。

## 5.2 元数据补丁的表达方式

建议补丁按字段级描述，而不是传一整份目标端完整元数据快照。

概念上可以支持以下操作：

- set：将某字段设置为指定值
- clear：清空某字段
- replace：替换多值字段
- merge：对某些可合并字段执行合并

适合纳入第一阶段的字段：

- title
- artist
- album
- albumArtist
- trackNumber
- discNumber
- genre
- year
- comment

暂不建议在第一阶段纳入：

- cover art
- lyrics
- 自定义 tag
- 复杂二进制附加块

## 5.3 目标端执行语义

设备 B 收到 patch 后，不应 blind write，而应执行以下流程：

1. 校验请求来源与授权
2. 定位目标文件
3. 校验文件当前状态是否仍满足请求前提
4. 执行 tag 写入
5. 返回结果
6. 刷新本地缓存、索引或预览数据

返回结果建议能区分：

- success
- file_not_found
- file_changed_since_preview
- unsupported_format
- permission_denied
- tag_write_failed
- partial_applied

## 6. 为什么“冲突项”是最适合的第一阶段范围

如果一开始就做“任意远端文件 tag 编辑”，范围会迅速扩大到：

- 普通浏览态编辑
- 批量编辑
- 非冲突文件编辑
- 复杂字段支持
- 权限与审计扩展

而如果第一阶段只聚焦“冲突项”，则范围更可控：

- 只对已识别的冲突项开放操作
- 只处理用户刚刚审阅过的差异
- 只下发少量、明确、可回显的字段 patch
- 失败时可直接要求重新拉取冲突详情

因此，它非常适合作为远端 tag 写入能力的最小可行入口。

## 7. 关键风险与约束

### 7.1 一致性风险

设备 A 看到设备 B 的 tag 后，用户可能在若干秒或若干分钟后才点击应用。这期间 B 上的文件可能已经变化。

因此请求中最好带上预期前置状态，例如：

- file size
- modified time
- content hash
- metadata fingerprint / revision

设备 B 在写入前校验这些信息；如果不匹配，则拒绝写入并要求重新获取最新状态。这本质上是乐观并发控制。

### 7.2 格式与写入能力差异

并不是所有音频格式都能稳定支持相同的 tag 写入能力。不同格式、不同底层库可能存在差异，例如：

- MP3 / ID3
- FLAC / Vorbis Comment
- M4A / MP4 atom
- WAV / APE / RIFF INFO

因此协议层需要允许 B 返回：

- 该格式支持读取但不支持写入
- 仅支持部分字段写入
- 写入后字段会发生规范化

### 7.3 路径不应作为唯一身份标识

如果整个 patch 链路完全依赖路径，未来很容易在重命名、移动或内容替换场景下出现错误命中。因此需要尽早引入更稳定的 file identity。

### 7.4 安全边界

哪怕只是修改 tag，这仍然属于远程写操作。设备 B 至少需要：

- 设备级信任校验
- 请求鉴权
- 仅允许修改库内文件
- 字段白名单
- 文件类型白名单
- 最低限度的操作记录或审计信息

## 8. 推荐的产品与架构表述

为了避免后续设计偏移，建议统一使用以下表述：

- 不是“设备 A 远程编辑设备 B 的文件”
- 而是“设备 A 向设备 B 提交 metadata patch，由设备 B 本机代理执行 tag 写入”

这个定义会直接影响：

- 协议结构
- 权限模型
- UI 文案
- 错误处理
- 冲突回滚与重试策略

## 9. 建议的分阶段落地顺序

### Phase 1：仅支持冲突项写回

范围建议限制为：

- 仅对已识别冲突项发起 patch
- 仅支持核心 tag 字段
- 仅支持单文件 patch
- 写回后立即刷新该冲突项状态

这是最有价值且最容易控制范围的阶段。

### Phase 2：扩展到普通远端 metadata 修正

在普通远端浏览态下，也允许对目标端文件发起 tag 修改。

### Phase 3：支持批量 metadata patch

例如：

- 批量修正 albumArtist
- 批量补 genre
- 批量修正编号字段

该阶段会显著提升复杂度，应放到前两阶段稳定之后再评估。

## 10. 当前建议

如果当前目标是处理“冲突项”，那么该能力值得做，且建议按以下原则推进：

- 使用字段级 metadata patch
- 由目标端本机执行写入
- 请求中携带版本或指纹校验信息
- 对可修改字段做白名单控制
- 第一阶段仅支持核心 tag 字段
- 第一阶段只服务于冲突项

## 11. 一句话结论

“步骤 3 看到目标端文件 tag”完全可以扩展为“设备 A 对设备 B 提交冲突项 tag 补丁”，但实现上应当定义为：

- 远端 metadata patch
- 目标端代理写入
- 带前置状态校验
- 先从冲突项最小范围落地
